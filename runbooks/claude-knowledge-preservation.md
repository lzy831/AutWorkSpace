# Claude 会话知识保鲜指南

## 核心原则

> **不要把关键知识留在会话里等 compaction 来保留，要主动写入文件系统，让文件成为长期记忆。**

## Claude 提示词加载机制

不同来源的内容进入 prompt 的可靠性完全不同：

```
╔═══════════════════════════════════════════════════════════╗
║  每次对话 100% 进入 System Prompt（最可靠）                 ║
║  ┌─────────────────────────────────────────────────┐     ║
║  │ CLAUDE.md (项目根目录 .claude/CLAUDE.md)           │     ║
║  │ ClaudeWorkSpace/ (通过 CLAUDE.md 引用)            │     ║
║  └─────────────────────────────────────────────────┘     ║
╠═══════════════════════════════════════════════════════════╣
║  按需加载（Claude 判断相关时自动读取）                      ║
║  ┌─────────────────────────────────────────────────┐     ║
║  │ Memory 文件 (~/.claude/projects/.../memory/)      │     ║
║  │ Skill 文件                                        │     ║
║  └─────────────────────────────────────────────────┘     ║
╠═══════════════════════════════════════════════════════════╣
║  会话内 Compaction 摘要（不可靠，信息随轮次衰减）           ║
║  ┌─────────────────────────────────────────────────┐     ║
║  │ Resume 后的 compaction 摘要                       │     ║
║  │ - 近期对话：完整保留                               │     ║
║  │ - 中期对话：压缩为摘要                             │     ║
║  │ - 早期对话：层层压缩，只剩骨架，细节永久丢失          │     ║
║  └─────────────────────────────────────────────────┘     ║
╚═══════════════════════════════════════════════════════════╝
```

## 知识保鲜分层策略

### 第 1 层：CLAUDE.md — 每次交互必加载（最可靠）

**维护内容：**
- 当前进行中的任务（简短一句话，完成即删）
- 常用的运行命令、调试技巧
- 项目架构约定
- 重要的"踩坑"经验

**示例：**
```markdown
## 当前任务
- [ ] 完善 ethernet suspend/resume 的 step/check
- [ ] 修复 router.py SSH 超时问题
```

### 第 2 层：ClaudeWorkSpace/decisions/ — 设计决策（⭐ 最推荐）

**命名格式：** `YYYY-MM-DD-<topic>.md`

**写入时机：**
- 和 Claude 讨论后做出的每个非显而易见的决策
- 每个阶段性任务完成后的总结
- 重要的踩坑记录

**实践方式：** 在对话中对 Claude 说：

> "把我们刚才关于 XXX 的讨论和决策，总结写入 `ClaudeWorkSpace/decisions/YYYY-MM-DD-<topic>.md`"

### 第 3 层：Memory — 持久化简短规则

**适合放的内容：**
- 简短的行为约束（1-2 句话）
- 关键环境参数
- 必须遵守的检查项

**不适合放的内容：**
- 长篇讨论 → 放 `decisions/`
- 规范文档 → 放 `conventions/` 或 `references/`

**示例：**
```
"每次修改 ethernet step/check 必须同步更新 inventory 文档"
"禁止直接修改 bin/ 下的二进制文件"
"DUT 的串口波特率是 921600"
```

### 第 4 层：ClaudeWorkSpace/ 文档目录

| 目录 | 用途 | 更新频率 |
|------|------|----------|
| `conventions/` | 架构、规范、格式约定 | 规范变更时 |
| `decisions/` | 设计决策、阶段总结 | 每次重要讨论后 |
| `references/` | step/check 清单、API 参考 | 新增/修改时 |
| `specs/` | 用例 spec YAML（规范源） | 用例变更时 |
| `runbooks/` | 操作手册、工作流程 | 流程变更时 |
| `environments/` | 拓扑、硬件、环境参数 | 环境变更时 |

## 日常操作习惯

### 习惯 1：阶段性"导出摘要"

每完成一个子任务后，让 Claude 将讨论要点写入 decisions：

> "把我们刚才关于 XXX 的讨论和决策，总结写入 `ClaudeWorkSpace/decisions/YYYY-MM-DD-<topic>.md`"

### 习惯 2：更新当前任务

在 CLAUDE.md 中维护 `## 当前任务` 章节，开始/完成任务时更新。每次新会话 Claude 能立即知道上下文。

### 习惯 3：善用 references/ 做知识沉淀

维护以下参考文档：
- `<module>-step-check-inventory.md` — step/check 完整清单
- `common-pitfalls.md` — 踩坑记录
- `debugging-notes.md` — 调试笔记

### 习惯 4：及时提交 decisions 到 git

讨论产生的 `decisions/` 文件应及时提交到 git，作为项目知识资产沉淀。

## 知识保鲜清单

| 知识类型 | 放哪里 | 何时写 |
|----------|--------|--------|
| 当前在做什么任务 | `CLAUDE.md` 当前任务章节 | 开始/完成任务时更新 |
| 设计决策 & 为什么这样做 | `decisions/YYYY-MM-DD-<topic>.md` | 每次重要讨论后立即写 |
| 规范 & 约定 | `conventions/<topic>.md` | 确立规范时写 |
| 简短行为约束 | `Memory` | 发现需要持续遵守的规则时 |
| step/check/API 清单 | `references/<topic>-inventory.md` | 新增/修改时更新 |
| 踩坑 & 调试经验 | `references/common-pitfalls.md` | 解决问题后立即写 |
| 操作流程 | `runbooks/<topic>.md` | 流程确立/变更时 |

## 注意事项

1. **别靠 compaction 记住任何重要决策** — 主动写入文件
2. **长会话定期做摘要导出** — 不要等会话结束才整理
3. **文件即文档** — decisions/ 下的 markdown 文件本身是最可靠的记录
4. **及时清理** — CLAUDE.md 中的"当前任务"完成后立即删除，避免误导后续会话
